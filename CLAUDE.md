## 技術スタック

- 静的サイト（HTML + CSS + Vanilla JS、フレームワーク無し）
- ホスティング: Vercel（mainブランチ自動デプロイ、feature branchはプレビューURL）
- データ: `brawlers.js`（現状1ファイルに全キャラ情報、要分割タスク有り）
- ビルドツール: なし（`package.json` はあるが現状ビルド不要）

---

## やり取りのルール

1. **言語**: 日本語でやり取りする
2. **プランモード徹底**: Research → Plan → 承認 → Build → Test の順
3. **重要な変更は事前確認**: ファイル削除、ファイル分割、データ構造変更、依存追加、新規ライブラリ導入などは必ず人間に確認を取る
4. **軽微な変更は自走OK**: タイポ修正、コメント追加、READMEの軽微な更新、コードフォーマット程度ならそのまま実行
5. **不明な時は質問**: 推測で進めず、選択肢を提示して聞く
6. **進捗の可視化**: 大きなタスクは TODO リストで分解して見せる

---

## Git運用ルール

- **mainブランチには直接pushしない**
- 作業は feature branch で行う
  - 命名規則: `feat/xxx`（新機能）、`fix/xxx`（バグ修正）、`refactor/xxx`（リファクタ）、`docs/xxx`（ドキュメント）
- featureブランチをpushすると、Vercelが自動でプレビューURLを生成する
- そのプレビューURLで動作確認 → OKなら PR を作成 → 人間が GitHub UI で merge
- **緊急時の例外**: 人間から「直接mainに入れて」と明示指示があった場合のみ直push可

---

## コーディング規約

- **ファイル命名**: kebab-case（例: `brawler-card.js`）
- **インデント**: スペース2つ
- **コメント**: 日本語OK、関数の役割は1行で説明
- **既存スタイルを優先**: 新ルールを持ち込む前に既存コードに合わせる

---

## やってはいけないこと

- `node_modules/`, `.env`, `_archive/` 配下の編集
- main への直push（人間の明示許可があれば例外）
- 動作確認なしでの大規模変更push
- アンケート結果（Google Forms）以外を根拠にした「ユーザーが欲しがってる」発言
- 既存ユーザーが使ってる機能を予告なく削除・変更すること

---

## やってほしいこと

- 作業前に関連ファイル全部読んで構造を把握
- 大きな変更はTODOリストで分解して見せる
- Vercelプレビューで動作確認の手順を毎回提示
- 進捗を `docs/CHANGELOG.md` に追記（後で振り返り用）
